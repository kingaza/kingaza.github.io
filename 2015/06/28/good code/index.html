好的代码

  • 软件开发

什么是好的代码？刚刚看了篇文章，作者通过咨询65个开发人员这一个问题，得到的答案是“好的代码是可读的，可理解的，覆盖了自动化测试的，不过于复杂，并且能办好我们需要它做的事情”。

这篇文章好代码的科学定义发表在51cto上的，估计是转载的。文章里提到的调查柑橘不是那么权威：样本少了点，咨询的对象也只是他们某个职位的应聘者，虽然看上去都有5年及以上的工作经验。但这是这些程序员的亲身体会，结论也是很有意思。
文章里面把数据统计出来

除了可读性和可理解性之外，其他的答案差异很大。我自己很诧异为啥“可维护性”并没有被足够重视。这一点作者也提到，因为我们大多数人都有过维护别人代码的经历（或者一段时间以后维护自己的代码），并且很有可能度过了一段非常悲惨的日子。所以，我们期待更多的开发人员能够编写出可维护的代码。可能有的人假设代码可读，那么一定易于维护，所以就忽略了这一条。
然后，作者归纳出来“好的代码”的定义。总的来说，这个定义是有道理的。

然后，问题就变成这个定义的意义了。无可置疑，我们应该写干净的、易于理解的、足够简单的代码，但如果因此走向极端追求美丽优雅趋向完美，那实在得不偿失。正如Andrew Hunt(《The Pragmatic Programmer: from Journeyman to Master》的作者)所说的，

“你不用去写所谓完美的软件。不是禁止你，只是真的没有这个必要。好好想想，接受这个话。你需要明白完美的软件其实是不存在的。计算机发展到今天还没有人写的软件是完美的。你也不要自以为是地想去做第一个。除非你能接受这个事实，否则你最终只会是在浪费时间和精力，如果你想追求这个不可能实现的梦想。”

所以我们要关注的重点是：这些代码是否如愿运行——是否正确、实用和高效？是否能处理异常数据而不崩溃？——或者至少是否能做到即使失败也不会出问题？是否容易调试？是否能简单安全地改变？这些实实在在的措施才是成功与失败之间的分水岭。

在另一篇文章不要浪费时间去写所谓的完美代码中提到了精益开发的见解：

精益开发的核心思想是：不要浪费时间去做那些不重要的事情，包括写代码、重构、代码审查以及代码测试等多个方面。

只需要重构真正需要的部分就足够了——这也被Martin Flower称之为是机会主义的重构和有准备的重构。

关于代码审查也只需要专注于重要部分。这些代码是否正确？是否安全？是否可以运行？

不要在意风格（除非风格本身妨碍了我们的理解）。让IDE做主，格式化的照顾就ok了。我们不必去讨论代码还能不能更OO，也不必一定要遵循某种样式，喜欢与否也没有关系，是否能用更好的方式解决也不重要——除非是你在教新手，并且需要做一些指导作为代码审查的一部分。

测试也要挑关键的来。测试要覆盖主要途径和重要的异常情况。无论是大型测试还是小而集中的测试，无论是写在代码之前还是之后，只要能起到作用就成。

上述基本也是我最近的一些困扰和思考。事实上，同时作为一个程序员和团队管理者，我是在理想主义和实用主义之前摇摆，怎么取得平衡是一个很有技巧的问题。
就此整理一下思路。
